## Story

**Como** operador do painel,
**quero** que o botão `Salvar Tudo` diferencie checkpoint local de sincronização com GitHub/Firebase e valide a identidade Git antes de preparar arquivos,
**para que** eu não receba uma confirmação enganosa nem tenha alterações não revisadas automaticamente colocadas em stage quando a publicação não puder ocorrer.

## Contexto

O registro `SYNC-20260622-1782178811894` informa Firebase Hosting bem-sucedido, mas GitHub falhou no commit por ausência de `user.name` e `user.email`. O endpoint retornou sucesso local e a interface exibiu “Tudo salvo. Cloud: ERROR”, sem explicar o estado parcial. Como o fluxo chama `git add -A` antes do commit, a falha também deixou um grande conjunto de arquivos staged.

O reparo deve preservar checkpoints locais, nunca limpar stage existente e bloquear a sincronização automática antes de qualquer `git add` quando identidade Git ou stage prévio impedirem um commit seguro.

## Acceptance Criteria

1. O sincronizador GitHub verifica repositório, identidade Git (`user.name` e `user.email`) e stage existente antes de qualquer `git add`.
2. Identidade ausente ou arquivos previamente staged interrompem somente a etapa GitHub, com mensagem acionável e sem alterar o stage.
3. O endpoint `save-all` informa separadamente que o checkpoint local foi salvo e se a nuvem terminou com sucesso, parcial ou erro.
4. A interface só diz “Tudo salvo” quando GitHub/Firebase terminarem com sucesso; em outro resultado, informa checkpoint local e o estado individual de cada destino.
5. Firebase Hosting e GitHub remoto são auditados de forma somente leitura; nenhuma publicação ou push é feita sem solicitação específica do usuário.
6. Testes cobrem os guardrails de identidade/stage e o texto de resultado parcial; quality gates passam.

## Dev Notes

- Não usar `git reset`, `git restore` ou limpeza de stage para corrigir este incidente: o repositório está sujo e os arquivos podem pertencer ao usuário.
- A configuração de identidade pode ser explicada ao usuário, mas não deve ser inferida nem gravada sem sua autorização explícita.
- Um resultado cloud em erro não invalida o checkpoint local que já foi persistido; a mensagem precisa refletir essa distinção.
