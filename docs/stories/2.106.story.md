## Story

**Como** responsável pelo projeto Antigravity,
**quero** uma cópia local completa no disco D e um mecanismo que a atualize após sincronizações bem-sucedidas com GitHub ou Firebase,
**para que** eu tenha um espelho local recuperável e um registro claro das versões publicadas, sem expor dados privados no repositório ou no Hosting.

## Acceptance Criteria

1. O espelho completo do workspace é criado em `D:\Antigravity-SommersStore\workspace`, com arquivos versionados, não versionados e dados locais preservados no disco D.
2. O sincronizador de espelho é executável via CLI e registra destino, momento, revisão Git e resultado em `D:\Antigravity-SommersStore\sync-state.json`.
3. Uma sincronização realizada pelo painel atualiza o espelho quando GitHub ou Firebase terminarem com sucesso, e informa falha do espelho sem esconder o resultado individual dos destinos em nuvem.
4. Um hook local pós-push e um comando de deploy Firebase com espelho cobrem as operações manuais suportadas pelo projeto; um agendamento local funciona como salvaguarda periódica.
5. O processo não faz `git add`, commit, push, deploy ou pull por conta própria e não altera a política de arquivos privados do `.gitignore`.
6. Testes automatizados cobrem a configuração, os códigos de saída do espelhamento e a integração do estado de nuvem; lint, typecheck e testes passam.

## Dev Notes

- GitHub armazena somente arquivos efetivamente commitados; Firebase Hosting publica apenas `projects/loja-digital/out_deploy`. Portanto, equivalência de conteúdo entre os três destinos não é tecnicamente correta nem segura.
- O espelho no D é completo e local, incluindo os arquivos que devem permanecer fora do GitHub por privacidade. O fluxo não faz sincronização bidirecional automática nem `git pull`, para não sobrescrever trabalho local.
- A publicação no GitHub continua condicionada aos guardrails existentes de identidade e stage vazio. O deploy Firebase manual deve usar o comando npm fornecido; o painel já chamará o espelho ao concluir o deploy.
